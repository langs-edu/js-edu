JavaScriptには「等しい」を調べる演算子が2つあります。`==`と`===`。なぜ2つあるのか——そこには、言語の生い立ちが刻まれています。

まず現物を見てみましょう。

```js
console.log("1" == 1);          // true
console.log("1" === 1);         // false
console.log(null == undefined); // true
console.log(0 == "");           // true
```

`==`は、比べる前に型を変換します。文字列の`"1"`と数値の`1`なら、型を揃えてから比べるので`true`。一見親切ですが、その変換ルールはかなり込み入っていて、`0 == ""`がtrueになるといった、直感に反する結果も生みます。一方の`===`は、型が違えばその時点で`false`。変換をしない、率直な比較です。

歴史をたどると、`==`は1995年の言語誕生時からいた古参です。おおらかな時代のWebでは、フォームから来た`"1"`と数値の`1`をいちいち区別しないほうが手軽だろう、という発想でした。対する`===`は少し遅れて、1997年に言語仕様(ECMAScript第1版)がまとめられる過程で正式に定義され、各ブラウザにも順次実装されていきます。つまり「ゆるい比較だけでは困る」という反省が、標準化のころにはもう共有されていたわけです。

現代の答えは、はっきりしています。常に`===`を使う。この流れを決定づけた一人が、『JavaScript: The Good Parts』(2008年)の著者Douglas Crockfordでした。同書は`==`を「使うべきでない部分」として名指しし、以来「比較は`===`」は世界中のコーディング規約の定番になっています。

<Callout type="note" title="自分と等しくない値">
ちなみに`NaN`だけは特別で、`NaN == NaN`も`NaN === NaN`もfalseになります。これは`==`と`===`の違いではなく、「NaNは何とも等しくない」と定めた浮動小数点数の国際標準に由来するふるまいです。NaNかどうかを調べたいときは`Number.isNaN()`を使います。
</Callout>

`==`は、古いコードを読むために知っておく演算子。自分で書くのは`===`。イコール1本の差は、30年分の教訓の差なのです。
