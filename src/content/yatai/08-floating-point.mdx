まずは、種も仕掛けもないところをお見せしましょう。

```js
console.log(0.1 + 0.2);
console.log(0.1 + 0.2 === 0.3);
```

```console
0.30000000000000004
false
```

0.1足す0.2が、0.3ちょうどにならない。初めて見ると手品のようですが、タネは明快です。

コンピュータは数を2進数(0と1の並び)で覚えています。ところが10進数の0.1は、2進数に直すと0.00011001100110011...と、同じ並びが無限に続く小数になります。10進数で1÷3が0.3333...と割り切れないのと同じ事情です。無限には覚えられないので、コンピュータはどこかで打ち切って丸めます。つまり0.1も0.2も、実はほんのわずかに違う値として記憶されていて、足し算のときにそのズレが顔を出す——それが手品の正体です。

大事なのは、これがJavaScriptの欠陥ではないことです。小数の扱い方はIEEE 754という1985年制定の国際標準で決められていて、PythonでもJavaでもCでも、同じ方式を使う言語なら同じことが起きます。実際にPythonで0.1 + 0.2を計算しても、まったく同じ0.30000000000000004が返ってきます。

では、どう付き合えばよいのでしょう。定石は3つです。第一に、小数同士を`===`でぴったり比較しない(差が十分小さいかどうかで判定します)。第二に、お金のように誤差が許されない計算は、最小単位の整数で行う(1.05ドルではなく105セントで数えます。日本円がはじめから整数なのは、ちょっとした幸運です)。第三に、人に見せるときだけ`toFixed`などで丸めて表示する。

夜店の手品は、タネを知ればただの手順です。0.1 + 0.2に驚かなくなったら、あなたはもう客席側ではなく、屋台側の人間ですよ。
