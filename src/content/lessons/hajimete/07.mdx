## エラーは、失敗ではなく返事

前回、`"abc" * 2`がエラーにならず`NaN`という値を返すことを見ました。でも、これまでのレッスンでも何度か、`const`への再代入のように、はっきりとエラーで止まる例にも出会っています。ここで、JavaScriptが出すエラーを整理して、正面から向き合っておきましょう。**エラーメッセージは、あなたを困らせるためのものではなく、「今、何が起きたか」を教えてくれる返事です。** 読み方さえ分かれば、こわいものではありません。

## SyntaxError——文としてすでに壊れている

いちばん早い段階で発覚するのが**SyntaxError**（構文エラー）です。丸かっこの閉じ忘れなど、コードが文としてそもそも成立していないときに起こります。次のコードは、実行できない例として示します。

```js nocheck
console.log("いらっしゃい"
```

閉じる丸かっこ`)`が足りません。これを実行しようとすると、次のようなメッセージが表示されます（実際の画面には、この上に問題の行を指し示す印なども表示されますが、ここでは要点だけを示しています）。

```console
SyntaxError: missing ) after argument list
```

SyntaxErrorは、**コードが1行も実行される前に**発覚するのが特徴です。「文として読めない」という、いちばん手前の段階のエラーだからです。

## ReferenceError——存在しない名前を呼んでいる

次によく出会うのが**ReferenceError**です。宣言していない変数名を使おうとしたときに起こります。

<CodeRunner client:visible storageKey="hajimete-07-a" expectError initialCode={`console.log(mikan);`} />

`mikan`という変数は、どこにも`const`や`let`で宣言していません。実行すると、次のメッセージが表示されます。

```console
ReferenceError: mikan is not defined
```

「`mikan`は定義されていません」——そのままの意味です。多くの場合、変数名のタイプミスか、宣言を書き忘れたことが原因です。

## TypeError——値の種類に合わない使い方をしている

3つ目が**TypeError**です。値の型に合わない操作をしようとしたときに起こります。

<CodeRunner client:visible storageKey="hajimete-07-b" expectError initialCode={`const kosuu = 5;
kosuu();`} />

`kosuu`は数値の`5`です。数値は関数ではないので、`kosuu()`のように丸かっこをつけて呼び出すことはできません。

```console
TypeError: kosuu is not a function
```

「`kosuu`は関数ではありません」。数値を関数として呼び出そうとした、という食い違いをそのまま教えてくれています。

## 静かに流れるundefinedと、そこから生まれるTypeError

ここまでの3つは、どれもエラーで止まる例でした。でも、JavaScriptには、エラーにならずに`undefined`が静かに流れていくケースもあります。両方を1つのコードで見てみましょう。

<CodeRunner client:visible storageKey="hajimete-07-c" initialCode={`let na;
console.log(na);`} />

`let na;`のように、`let`で名前だけ宣言して値を代入しないと、その変数には自動的に`undefined`という値が入ります。`console.log(na)`を実行してもエラーにはならず、**「まだ何も入っていない」ことを表す`undefined`という値**が静かに出力されます。ここまでは、前回見た`NaN`と同じ「エラーにならずに流れる」パターンです。

では、その`undefined`をさらに使おうとするとどうなるでしょうか。

<CodeRunner client:visible storageKey="hajimete-07-d" expectError initialCode={`let na;
console.log(na);
console.log(na.length);`} />

1行目はさっきと同じく`undefined`が出力されますが、2行目でついにエラーになります。

```console
TypeError: Cannot read properties of undefined (reading 'length')
```

「`undefined`のプロパティは読めません」。`na`の中身が`undefined`だったので、そこにさらに`.length`をつけて中身を読もうとしたところで、ようやく食い違いが表面化しました。**JavaScriptは、止まって教えてくれるとは限らない読み手です。** `undefined`や`NaN`は、発生した瞬間には教えてくれず、それを次に使おうとした瞬間に初めてエラーとして現れることがよくあります。

<Callout type="warn" title="つまずきポイント：エラーの発生場所と、原因の場所は違うことがある">
上の例で、実際にエラーになった行は3行目でしたが、根本の原因は2行目——`na`に何も値が代入されていなかったこと——にあります。エラーメッセージの行番号だけを見て「ここが悪い」と決めつけず、その値がどこから来たのかを、少し上の行までさかのぼって考える習慣をつけておくと、原因を見つけやすくなります。
</Callout>

## スタックトレースの読み方

エラーメッセージの下に、`at ...`で始まる行がいくつか続くことがあります。これを**スタックトレース**と呼びます。エラーが起きた場所から、そこを呼び出した場所へと、内側から外側に向かって順に並んだ記録です。行数が多くて圧倒されそうになりますが、慣れないうちは、いちばん上の1行——エラーの種類とメッセージそのもの——だけ読めれば、たいていは十分です。

このコースのCodeRunnerでも、ブラウザの開発者ツールのコンソール（レッスン2）でも、エラーが起きたときの基本的な表示のされ方は同じです。同じJavaScriptという言語が、同じルールでエラーを教えてくれています。

## このレッスンで分かったこと

- **SyntaxError**は文として成立していない、**ReferenceError**は存在しない名前を使った、**TypeError**は値の種類に合わない使い方をした、というサイン
- `undefined`や`NaN`は、発生した瞬間はエラーにならず、それを次に使おうとした瞬間にTypeErrorとして表面化することがある
- エラーメッセージの`at ...`の並びはスタックトレース。慣れないうちは、いちばん上の1行だけ読めれば十分
- CodeRunnerでも、ブラウザの開発者ツールでも、エラーの読み方は同じ
