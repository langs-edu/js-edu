## エラーで、プログラムを止めたくないとき

前回、壊れたJSON文字列を`JSON.parse`に渡すと、`SyntaxError`が発生してプログラムが止まりました。でも、いつもそこで止まってほしいわけではありません。「壊れたデータが来ることもある」と分かっているなら、そのエラーを受け止めて、プログラム自体は続けたい場面もあります。

そのための書き方が**try/catch**です。

## try/catchで、エラーを受け止める

<CodeRunner
  client:visible
  storageKey="sekkei-07-a"
  initialCode={`const broken = '{name: "ラムネ", price: 200}';

try {
  const data = JSON.parse(broken);
  console.log(data);
} catch (error) {
  console.log("読み込みに失敗しました:", error.message);
}

console.log("ここまで実行が続く");`}
/>

`try { ... }`の中で実行してみて、エラーが起きたら`catch (error) { ... }`にその場で移ります。`error`には、起きたエラーを表す**エラーオブジェクト**が渡り、`error.message`でエラーメッセージの文字列だけを取り出せます。ポイントは、`catch`のブロックを抜けたあと、プログラムがそのまま続くことです。`console.log("ここまで実行が続く")`が、ちゃんと実行されています。`try`で囲まなければ、レッスン6で見たとおり、ここでプログラムは止まっていました。

## finally——うまくいっても失敗しても、必ず実行する

`try`・`catch`に続けて、`finally`というブロックを追加できます。成功しても失敗しても、必ず実行される後片付け用のブロックです。

<CodeRunner
  client:visible
  storageKey="sekkei-07-b"
  initialCode={`function loadData(text) {
  try {
    return JSON.parse(text);
  } catch (error) {
    console.log("エラー:", error.message);
    return null;
  } finally {
    console.log("読み込み処理を終了します");
  }
}

console.log(loadData('{"name": "ラムネ"}'));
console.log(loadData('{name: "ラムネ"}'));`}
/>

1回目の`loadData`はエラーなしで成功しますが、それでも`読み込み処理を終了します`が表示されています。2回目は`catch`でエラーを受け止めたあとにも、同じく`読み込み処理を終了します`が表示されます。`finally`は「成功したから」「失敗したから」に関係なく、**必ず**実行されます。しかも、`try`や`catch`の中に`return`があっても、関数が実際に呼び出し元へ値を返すより前に、`finally`が実行されます。ファイルを閉じる、接続を切るなど、「後片付けは必ずやりたい」処理に向いています。

## throw——自分からエラーを起こす

ここまでは、JavaScriptが自動的に起こすエラーを受け止めるだけでした。でも、自分の書いたコードの中で「これは明らかにおかしい」と分かった場面では、自分からエラーを起こすこともできます。それが`throw`です。

レッスン5の`Yatai`クラスの`sell`メソッドに、チェックを追加してみます。

<CodeRunner
  client:visible
  storageKey="sekkei-07-c"
  initialCode={`class Yatai {
  constructor(name, price, stock) {
    this.name = name;
    this.price = price;
    this.stock = stock;
  }

  sell(count) {
    if (count <= 0) {
      throw new Error("個数は1以上にしてください");
    }
    if (count > this.stock) {
      throw new Error("在庫が足りません");
    }
    this.stock -= count;
    return this.price * count;
  }
}

const ramune = new Yatai("ラムネ", 200, 3);

try {
  console.log(ramune.sell(5));
} catch (error) {
  console.log("販売できませんでした:", error.message);
}

console.log("在庫:", ramune.stock);`}
/>

`throw new Error("メッセージ")`は、「ここでエラーを発生させる」という文です。`sell`の中で在庫より多い数を売ろうとしたので、`throw`が実行され、その場で`sell`の実行が止まり、呼び出し元の`catch`まで一気に飛びます。最後の`在庫: 3`が示すとおり、在庫は減っていません——チェックのおかげで、おかしな状態になる前に止まりました。

<Callout type="warn" title="つまずきポイント：throwを、catchせずに実行すると">
`throw`されたエラーを、どの`try/catch`も受け止めなければ、レッスン1・3・5で見た自動発生のエラーと同じように、プログラムはそこで止まります。

```js expect-error
class Yatai {
  constructor(name, price, stock) {
    this.name = name;
    this.price = price;
    this.stock = stock;
  }

  sell(count) {
    if (count <= 0) {
      throw new Error("個数は1以上にしてください");
    }
    this.stock -= count;
    return this.price * count;
  }
}

const ramune = new Yatai("ラムネ", 200, 10);
console.log(ramune.sell(0));
```

実行すると、次のようなエラーになります（実際にはこの下に、どのメソッドの何行目で起きたかを示すスタックトレースも表示されますが、ここではエラーの種類とメッセージだけ示します）。

```console
Error: 個数は1以上にしてください
```

`throw`は「わざとエラーを起こす」だけで、それ自体は`catch`されなければ止まります。**止めたいときはそのまま`throw`だけ、続けたいときは呼び出す側で`try/catch`を用意する**——この2つはセットで使います。
</Callout>

## なぜ、自分からthrowするのか

コース2で、JavaScriptは「止まって教えてくれるとは限らない」言語だと学びました。`"abc" * 2`は`NaN`に、存在しない配列の添字は`undefined`になり、エラーにはなりません。**JSは、放っておくと、おかしな値のまま静かに流れ続けることが多い言語です。**

だからこそ、自分のコードにとって「これ以上進んだらおかしくなる」という条件が分かっているなら、`throw`で明示的に止める価値があります。`sell`に負の個数を渡しても、`throw`がなければ在庫がマイナスになったまま、何事もなく計算が進んでしまいます。`throw`は、静かに流れていくはずだった間違いを、その場で立ち止まらせるための道具です。

## このレッスンで分かったこと

- `try { ... } catch (error) { ... }`で、エラーを受け止めてプログラムを止めずに続けられる
- `error.message`で、エラーメッセージの文字列を取り出せる
- `finally { ... }`は、成功・失敗にかかわらず必ず実行される後片付け用のブロック
- `throw new Error("メッセージ")`で、自分からエラーを起こせる。`catch`されなければ、そこでプログラムは止まる
- JavaScriptは、`NaN`や`undefined`のように、おかしな値のまま静かに流れ続けることが多い言語。だからこそ、「これ以上進んだらおかしい」という条件には、自分から`throw`して止める価値がある
