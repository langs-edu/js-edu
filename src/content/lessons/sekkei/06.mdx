## オブジェクトのままでは、運べない

ここまでのオブジェクトや`class`のインスタンスは、プログラムが動いている間、メモリの中に存在しています。でも、それをファイルに保存したり、別のプログラムに送ったりしたいときは、オブジェクトのままではいけません。保存も通信も、扱えるのは**文字列**（や、それに近いバイト列）だけだからです。

「JavaScriptのオブジェクトを、文字列に変換する共通のルール」——それが**JSON**（JavaScript Object Notation）です。

## JSON.stringifyとJSON.parse

オブジェクトを文字列にするのが`JSON.stringify`、文字列をオブジェクトに戻すのが`JSON.parse`です。

<CodeRunner
  client:visible
  storageKey="sekkei-06-a"
  initialCode={`const yatai = { name: "りんご飴", price: 300 };

const json = JSON.stringify(yatai);
console.log(json);
console.log(typeof json);

const restored = JSON.parse(json);
console.log(restored);
console.log(restored.name);`}
/>

`JSON.stringify(yatai)`は、オブジェクト`yatai`を、`'{"name":"りんご飴","price":300}'`という**文字列**に変換します。`typeof json`が`"string"`であることからも、これがもうオブジェクトではなく、ただの文字列だと分かります。`JSON.parse(json)`は、その逆——文字列を読み取って、もとと同じ形のオブジェクトを作ります。

<Callout type="note" title="restoredは、yataiとは別の新しいオブジェクト">
`JSON.parse`が作る`restored`は、`yatai`と見た目は同じでも、まったく別の新しいオブジェクトです（`restored === yatai`は`false`になります）。`JSON.stringify`と`JSON.parse`の組は、オブジェクトそのものを運ぶのではなく、オブジェクトの中身を文字列に写し取り、その文字列から同じ中身のオブジェクトを新しく組み立て直しています。
</Callout>

## 見やすく整形する

`JSON.stringify`の第2引数と第3引数を使うと、見やすく整形できます。

<CodeRunner

  initialCode={`const yatai = { name: "りんご飴", price: 300, stock: 5 };

/>

第2引数の`null`は「すべてのプロパティをそのまま含める」という意味、第3引数の`2`は「1段のインデントに半角スペース2つ使う」という意味です。設定ファイルやログとして人間が読むときは、この整形つきの形がよく使われます。

## JSONは、JSのオブジェクトそのものではない

JSONという名前は「JavaScript Object Notation」の略ですが、JSONはJavaScriptのオブジェクトリテラルと**まったく同じ**ではありません。いくつか、はっきりした違いがあります。

- キーは、必ず**ダブルクォートで囲んだ文字列**でなければならない（JSのオブジェクトリテラルでは省略できる）
- 値になれるのは、文字列・数値・真偽値・`null`・配列・オブジェクトだけ。**関数は値にできない**
- `undefined`もJSONの値にはなれない
- 最後の要素のあとにカンマ（トレイリングカンマ）を置けない
- コメントを書けない

<Callout type="warn" title="つまずきポイント：JSっぽく書いた文字列は、JSONとして読めない">
JSのオブジェクトリテラルの見た目のまま、キーにクォートをつけずに書いた文字列を`JSON.parse`に渡すと、エラーになります。

```js expect-error
const broken = '{name: "りんご飴", price: 300}';
console.log(JSON.parse(broken));
```

```console
SyntaxError: Expected property name or '}' in JSON at position 1 (line 1 column 2)
```

JSONの世界では、キーの`"name"`のダブルクォートは省略できません。「JSのオブジェクトリテラルとして正しい」ことと、「JSONとして正しい」ことは、別の基準です。
</Callout>

## classのインスタンスをJSONにすると、何が残る？

レッスン5で作った`Yatai`クラスのインスタンスを、`JSON.stringify`にかけてみます。

<CodeRunner
  client:visible
  storageKey="sekkei-06-c"
  initialCode={`class Yatai {
  constructor(name, price, stock) {
    this.name = name;
    this.price = price;
    this.stock = stock;
  }

  describe() {
    return \`\${this.name}（\${this.price}円、残り\${this.stock}個）\`;
  }
}

const appleCandy = new Yatai("りんご飴", 300, 5);
const json = JSON.stringify(appleCandy);
console.log(json);

const restored = JSON.parse(json);
console.log(restored);
console.log(restored instanceof Yatai);
console.log(typeof restored.describe);`}
/>

`JSON.stringify(appleCandy)`の結果には、`name`・`price`・`stock`は含まれていますが、`describe`メソッドは含まれていません。**JSONに残るのは、インスタンス自身が持つデータ（プロパティ）だけで、メソッドは含まれません。** さらに、`JSON.parse`で戻ってきた`restored`は、`Yatai`のインスタンスではなく、**ただのオブジェクト**です（`restored instanceof Yatai`は`false`、`restored.describe`は`undefined`）。「同じ形に見えるデータ」に戻るだけで、「同じクラスのインスタンス」には戻らない——これは覚えておく価値のある、誠実な事実です。

<Callout type="yorimichi" title="なぜJSONが大事なのか">
JSONが便利なのは、JavaScriptだけの都合ではありません。Python・Java・Rustなど、ほとんどのプログラミング言語がJSONを読み書きできます。設定ファイル、Webのやり取り、アプリ同士のデータの受け渡し——JSONは、言語をまたいだ「共通語」として、いまや当たり前のように使われています。詳しい歴史は、[🏮 屋台のこぼれ話「JSONの発明」](../../../yatai/11-json/)で。
</Callout>

## このレッスンで分かったこと

- `JSON.stringify(値)`は、オブジェクトや配列を文字列に変換する。`JSON.parse(文字列)`は、その逆
- `JSON.stringify(値, null, 数)`で、見やすく整形した文字列を作れる
- JSONは、キーに必ずダブルクォートが必要など、JSのオブジェクトリテラルより厳密なルールを持つ。関数や`undefined`は値にできない
- `class`のインスタンスを`JSON.stringify`すると、プロパティは残るがメソッドは含まれない
- `JSON.parse`で戻ってくるのは、もとのクラスのインスタンスではなく、ただのオブジェクト
- JSONは、言語をまたいだデータの共通語として、保存や通信で広く使われている
