## 記憶を、ページを閉じても残るものに

前回の`storage.js`は、`let books = []`という、ただの変数でした。ページを閉じたり再読み込みしたりすれば、この記録はきれいさっぱり消えます。今日のゴールは、本の記録をブラウザに保存し、次に開いたときも読み込めるようにすることです。

使うのは、ブラウザが持っている`localStorage`という道具です。

## localStorageは、文字列だけのキーと値の保存箱

`localStorage`は、`キー`と`値`の組を、ブラウザの中に保存しておける仕組みです。使い方はとても単純で、2つのメソッドだけで足ります。

- `localStorage.setItem(キー, 値)` — 保存する
- `localStorage.getItem(キー)` — 取り出す。保存されていなければ`null`

ブラウザの開発者ツールのコンソールで試すと、こう動きます。

```console
localStorage.setItem("hondana-books", JSON.stringify([{ title: "三体", author: "劉慈欣", rating: 5, review: "スケールがすごい" }]));
console.log(localStorage.getItem("hondana-books"));
console.log(localStorage.getItem("nothing-here"));
```

```console
[{"title":"三体","author":"劉慈欣","rating":5,"review":"スケールがすごい"}]
null
```

ここで大事な制限が1つあります。**`localStorage`に保存できるのは、文字列だけです。** 数値や真偽値を渡すと自動的に文字列に変換されますが、配列やオブジェクトをそのまま渡すと、`"[object Object]"`のような、中身の分からない文字列になってしまいます。

## JSON.stringify/parseの、伏線回収

配列やオブジェクトを保存したいなら、まず文字列に変換する必要があります。ここで、コース4で学んだ`JSON.stringify`と`JSON.parse`が働きます。

  { title: "三体", author: "劉慈欣", rating: 5, review: "スケールがすごい" },

console.log(json);
console.log(typeof json);

const revived = JSON.parse(json);
console.log(revived);

/>

`JSON.stringify(books)`で、配列をまるごと1本の文字列にします。これなら`localStorage.setItem`にそのまま渡せます。読み込むときは`JSON.parse(json)`で、文字列を配列とオブジェクトに戻します。レッスン1で見た「classはJSON化するとメソッドが失われる」という話を思い出してください。hondanaの本はもともとメソッドを持たないプレーンオブジェクトなので、この行き来で失われるものは何もありません。

## storage.jsを、本物にする

`loadBooks`と`saveBooks`という名前と役割は変えず、中身だけを`localStorage`を使う本物の実装に置き換えます。

```js title="storage.js"
// storage.js
// 本の一覧を、ブラウザのlocalStorageへ保存・読み込みする係。

const STORAGE_KEY = "hondana-books";

export function loadBooks() {
  const json = localStorage.getItem(STORAGE_KEY);
  if (json === null) {
    return [];
  }
  return JSON.parse(json);
}

export function saveBooks(books) {
  localStorage.setItem(STORAGE_KEY, JSON.stringify(books));
}
```

<Callout type="note" title="hondana.jsは、1行も変えていません">
`loadBooks`と`saveBooks`という名前、受け取る引数、返す値の形——この「外から見た約束」がレッスン1と変わっていないので、`hondana.js`は前回のままで動きます。中の仕組みを「その場限りの記憶」から「本物の保存」に差し替えても、呼び出す側は何も知らなくていい。これが、モジュールを分けておく利点の1つです。
</Callout>

`STORAGE_KEY`という定数に`"hondana-books"`という名前を1か所にまとめているのにも意味があります。この文字列を`localStorage`のキーとして毎回タイプすると、どこかで綴りを間違えたときに気づきにくくなります。1か所にまとめておけば、間違えようがありません。

## ブラウザに保存される、ということの意味

`localStorage`は便利ですが、いくつか誠実に知っておくべき限界があります。

- **そのブラウザ・そのオリジン(サイト)だけ**に保存されます。別のブラウザや、シークレットウィンドウ、別のパソコンからは見えません
- ブラウザの設定で「サイトデータを消去」すると、保存した記録も一緒に消えます
- 保存できる容量には上限があります(ブラウザによって違いますが、数MB程度が目安です)。hondanaのような小さなテキスト記録なら、まず心配いりません

「サーバーに預ける」のではなく「この端末のこのブラウザに置いておく」——localStorageはそういう保存です。この性質は、次のレッスンで一覧を作るときにも関わってきます。

`localStorage.getItem`は、保存されていないキーに対して`null`を返します。もし`if (json === null)`のチェックを忘れて`JSON.parse(null)`を呼んだら、どうなるでしょうか。

<CodeRunner
  client:visible
  storageKey="koubou-02-b"
  initialCode={`const stored = null; // まだ何も保存されていないときを想定
const books = JSON.parse(stored);
console.log(books);`}

エラーにならず、`null`という値がそのまま返ってきます。**`JSON.parse`は、渡された`null`を文字列`"null"`として解釈し、それをパースした結果の`null`という値を返す**からです。ここまでは何も起きません。危険なのはこのあとです。

<CodeRunner

  storageKey="koubou-02-c"
  expectError
  initialCode={`const stored = null;
const books = JSON.parse(stored);
console.log(\`\${books.length}冊\`);`}
/>

```console
TypeError: Cannot read properties of null (reading 'length')
```

<Callout type="warn" title="つまずきポイント: JSON.parse(null)は、静かに通り抜ける">
`books`を配列として使おうとした、この行で初めてエラーになります。「まだ何も保存されていないとき」を`if (json === null) return [];`で先に処理しておかないと、エラーの発生する場所が、原因から離れたところにずれてしまいます。コース1で見た「エラーにならずに流れていく」というJavaScriptの性格が、ここでも顔を出しています。
</Callout>

## このレッスンで分かったこと

- `localStorage`は、文字列のキーと値だけを保存できる、ブラウザの保存箱
- `JSON.stringify`で配列やオブジェクトを文字列にしてから保存し、`JSON.parse`で文字列から戻す
- 保存はそのブラウザ・そのオリジンだけに閉じている。消えることもある、という限界を知っておく
- `JSON.parse(null)`はエラーにならず`null`を返す。配列として使おうとした、もっと後の行で初めてエラーになる
- `loadBooks`・`saveBooks`という名前と役割を変えずに中身だけ差し替えられたのは、モジュールで境界を分けておいた効果
